This project demonstrates the complete logic and user flow of a secure One-Time Password (OTP) authentication system. It is broken into two main parts:

  1. index.html (The Logic Illustrator): A 3-box dashboard that visualizes the "behind-the-scenes" logic of OTP generation, storage, and verification.
  2. auth-page.html (The User-Facing App): A realistic, single-page application that simulates the complete user experience of logging in with an email/phone number and a one-time code.

1. What is an OTP?

A One-Time Password (OTP) is a password that is valid for only one login session or transaction, on a single device. It is a form of strong authentication because it is generated randomly and expires quickly, making it incredibly difficult for an attacker to guess or reuse.

OTPs are the foundation of Two-Factor Authentication (2FA).

2. The OTP Implementation & Verification Flow

This project perfectly illustrates the three main stages of an OTP lifecycle.

Step 1: Generation

When a user wants to log in or perform a secure action (like a bank transfer), they first request an OTP.

  • How it works: The user enters their identifier (like an email or phone number) and clicks "Send OTP."
  • On the Server: The server generates a cryptographically random, 6-digit numerical code.
  • In this Project:
    • index.html: Clicking the "Generate / Reset OTP" button in Box 1 simulates this.
    • auth-page.html: Submitting your email on the first screen triggers this.

Step 2: Storage & Delivery

Once generated, the server must store the code and send it to the user.

  • How it works: The server saves the code (e.g., 123456) in its database, links it to the user, and sets a short expiry time (e.g., 60 seconds). It then sends this code to the user via an Email or SMS.
  • In this Project:
    • index.html: Box 2 (Simulated Storage) acts as our "database." It shows you the code and starts the 60-second timer.
    • auth-page.html: The app simulates sending by logging the OTP to the browser console, then immediately shows the verification screen.

Step 3: Verification

This is the final step where the user proves they have the code.

  • How it works: The user types the 6-digit code they received into the application. They click "Verify." The server then checks:
    1. Is this code the exact same one I have stored for this user?
    2. Has the 60-second timer expired?
    3. (Security) Has the user failed too many times (e.g., 3 attempts)?
  • In this Project:
    • index.html: Box 3 (Verify OTP) performs this check against the value in Box 2.
    • auth-page.html: The 6-digit input box does the same check against the code stored in the console.

If all checks pass, the user is authenticated. If any check fails (wrong code, expired, too many attempts), the verification fails, and the user must start over.

3. The Importance of OTP

The "why" is the most critical part. Passwords alone are not secure.

  • Passwords Can Be Stolen: Hackers can steal passwords through phishing, database leaks, or guessing.
  • OTP Solves This: OTP implements Two-Factor Authentication (2FA). This means a user must provide two different types of credentials to log in:
    1. Factor 1: Something you KNOW (Your password)
    2. Factor 2: Something you HAVE (Your phone, which receives the OTP)

A hacker might steal your password, but it's useless without the second factorâ€”the OTP that is generated in real-time and sent only to your physical device.

This small, 6-digit code is one of the most effective tools we have to protect against account takeovers, secure financial transactions, and prevent unauthorized access.

4. Advanced Security Features in This Project

This simulation also includes critical security features found in real-world systems:

  • OTP Expiry (60s Timer): Ensures that an old, used, or stolen code cannot be used again later.
  • Verification Attempt Limits (3 Strikes): Prevents a "brute-force" attack, where a hacker rapidly tries to guess the code (e.g., 000000, 000001, 000002...).
  • Resend Rate Limiting (30s Delay): Stops a user (or attacker) from spamming the "Send OTP" button. This is crucial as every SMS or Email sent costs the company money and can be used as an abuse vector.
